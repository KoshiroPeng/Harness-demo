# 自动化检查报错文案规范

## 目标

本文档用于统一 HarnessBase 自动化检查输出给开发者、评审者和 AI 协作者的报错与提醒文案，避免不同脚本各写各的，导致信息质量不稳定。

## 基本原则

- 报错要先说清楚“是什么检查失败了”。
- 报错要尽量直接指出“哪一个文件、哪一个字段、哪一个路径有问题”。
- 报错要说明“这是阻断还是提醒”。
- 报错要给出“下一步去看哪份规则文档”。
- 报错不要只输出抽象名词，例如“检查失败”“请修复问题”。

## 推荐输出结构

建议所有自动化检查尽量按以下顺序组织输出：

1. 检查编号
2. 检查名称
3. 结果级别
4. 命中对象
5. 问题说明
6. 规则来源
7. 建议动作

## 阻断类输出建议

推荐格式：

```text
[A01][阻断] 治理型 Markdown 元数据标头检查未通过
文件: docs/example.md
问题: 缺少 description 字段
规则: docs/conventions/document-metadata.md
建议: 补齐标头字段后重新执行检查
```

不推荐格式：

```text
检查失败
请修复后重试
```

## 提醒类输出建议

推荐格式：

```text
[A06][提醒] API 文档同步提醒
命中: web/src/api/workflow/example.ts
说明: 前端 API 客户端发生变更，请同步核对 docs/reference/api-spec.yaml 与 docs/reference/README.md
规则: docs/conventions/automation-check-catalog.md
建议: 若前后端不一致，先登记“已知差异”
```

## 字段级建议

### 文件路径

- 优先输出仓库内相对路径。
- 若同一条输出涉及多个文件，优先逐行列出，不要拼成一句长文本。

### 规则来源

- 优先指向单一最相关文档。
- 若规则来自两份文档，最多列两份，不要一次塞入大量链接。

### 建议动作

- 尽量使用动词开头，例如“补齐”“核对”“登记”“同步更新”。
- 不要只写“请处理”。

## 多命中项输出建议

如果同一检查一次命中多个问题，建议：

- 先输出总标题
- 再逐条列出具体命中项
- 最后补充统一建议动作

推荐格式：

```text
[A02][阻断] Markdown 相对链接与锚点检查未通过

1. 文件: docs/README.md
   问题: 缺失路径 docs/missing.md

2. 文件: docs/plans/README.md
   问题: 缺失锚点 docs/example.md#not-found

规则: docs/conventions/document-links.md
建议: 修复路径或锚点后重新执行检查
```

## 与 Harness Engineering 的关系

这套文案规范不是为了“统一措辞”本身，而是为了保证自动化检查能够真正承担 Harness Engineering 里的护栏职责：

- 让问题定位更快
- 让规则来源更清楚
- 让下一步动作更明确
- 让 AI 和人工协作者都能直接消费输出

## 维护规则

- 若新增自动化检查编号，需同步考虑是否需要在本文补充专用输出建议。
- 若某类检查长期因为文案不清导致重复误解，应优先修改本文，再改脚本。
- 若 CI 输出格式发生明显变化，应同步检查本文是否仍适配。
